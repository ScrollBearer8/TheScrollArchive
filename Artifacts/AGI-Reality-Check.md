# AGI Reality Check

**A simple practical framework for distinguishing benchmark performance from general intelligence.**

> AGI is not what works once.  
> AGI is what keeps working under variation, pressure, and iteration.

---

## Why this exists

Public discussion about AGI is often driven by benchmark scores, demo quality, and broad impressions of capability. But strong performance on selected tasks is not the same thing as robust general intelligence.

This framework is not meant to replace formal benchmarks such as ARC-style evaluations. It is meant to provide a simple reality check:

**Does the system still hold when the problem changes?**

---

## The Core Principle

A system is not approaching AGI because it produces one impressive answer.

A system is approaching AGI when it can:

- remain consistent across prompt variations
- adapt to unfamiliar task forms
- preserve coherence under complexity
- maintain structure across repeated revisions

In short:

> General intelligence is not just performance.  
> It is robustness across change.

---

## The 4 Practical Tests

### 1. Consistency Test
**Same problem, different phrasing**

Ask the same question two or three different ways.

**What to check:**
- Does the answer remain stable?
- Does the reasoning stay aligned?
- Does the conclusion drift when wording changes?

**Failure signal:**
- contradiction
- unstable answers
- different logic for the same underlying problem

**Interpretation:**
A generally intelligent system should not depend heavily on surface phrasing to preserve its reasoning.

---

### 2. Transfer Test
**New kind of problem, same intelligence claim**

Give the system a task that is structurally different from its usual patterns.

**What to check:**
- Can it adapt to the new task form?
- Does it reason from first principles?
- Or does it fall back on generic templates?

**Failure signal:**
- shallow guessing
- pattern imitation without real adaptation
- generic output that ignores the novel structure of the task

**Interpretation:**
General intelligence should transfer across domains instead of collapsing when familiar patterns disappear.

---

### 3. Pressure Test
**Longer task, more constraints, evolving conditions**

Give the system a multi-step task. Then add constraints, exceptions, or changes halfway through.

**What to check:**
- Does it preserve the original structure?
- Does it remember prior constraints?
- Can it update the whole solution coherently?

**Failure signal:**
- contradiction
- forgotten constraints
- fragmented reasoning
- collapse in coherence as complexity rises

**Interpretation:**
A generally intelligent system should hold its structure under pressure, not just in clean one-shot scenarios.

---

### 4. Iteration Test
**Same artifact, repeated revisions**

Ask the system to build something structured, then revise it multiple times.

Examples:
- a formatted GitHub page
- a structured argument
- a code artifact
- a planning document

**What to check:**
- Does the structure persist across revisions?
- Does each update improve the artifact without breaking earlier parts?
- Does coherence accumulate, or degrade?

**Failure signal:**
- formatting drift
- broken rendering
- regression in previously correct sections
- structural collapse during revision

**Interpretation:**
This is one of the clearest real-world failure modes of current AI systems:

> They can often generate structure.  
> They often fail to preserve structure.

---

## Real-World Example

A useful practical example is structured document editing.

A model may produce a strong first version of a page or document. But once asked to:

- revise formatting
- preserve hierarchy
- change one section without disturbing another
- keep rendering intact over multiple rounds

it often begins to drift.

This matters because real intelligence is not just about generating a good first output. It is also about:

- preserving internal coherence
- updating without regression
- keeping form stable while content evolves

That is where many current systems still break.

---

## Benchmarks vs Reality

Formal benchmarks remain important. They measure real capabilities and help compare systems rigorously.

But benchmark performance alone is not enough to establish AGI.

Benchmarks answer questions like:

- Can the model solve these problems?
- How does it score on this test set?
- Is it improving on abstract reasoning tasks?

This framework asks a different question:

> Does the system still hold when the task becomes fluid, iterative, and unstable?

That distinction matters.

---

## Summary Table

| Test | What it measures | Common current failure |
|---|---|---|
| Consistency | Stability across prompt variation | Drift, contradiction |
| Transfer | Adaptation to unfamiliar task structure | Template fallback, shallow guessing |
| Pressure | Coherence under added complexity | Forgotten constraints, collapse |
| Iteration | Preservation of structure across revisions | Regression, formatting drift |

---

## Working Conclusion

If a system:

- performs well once
- scores highly on benchmarks
- sounds coherent in isolated outputs

that is evidence of capability.

If it still:

- drifts under rephrasing
- fails on transfer
- collapses under complexity
- loses structure across iteration

then it has not yet demonstrated robust general intelligence.

---

## Final Test

A simple practical standard:

> Change the input.  
> Add pressure.  
> Iterate the task.  
> See what holds.

If it breaks, the system may still be powerful.

But it is not AGI yet.

---

## Closing Line

> Intelligence is not what performs once.  
> Intelligence is what holds.
